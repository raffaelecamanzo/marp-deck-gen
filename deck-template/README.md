 # MARP Deck Template and Workflow

This folder contains a MARP deck template, theme, and a Dockerized Taskfile workflow to preview and export your slides. It also references a set of skills that automate the endâ€‘toâ€‘end deck creation from a content definition.

## Folder structure
- `slides/deck.md`: Final MARP deck. Contains a frontpage and a closing slide; insert user slides between them.
- `.marp/themes/deck-theme.css`: Custom theme (`@theme deck-theme`).
- `definition/deck-definition.md`: Input content definition (sections, key messages, constraints).
- `planning/`: Planning artifacts generated by the workflow:
  - `planning/deck-plan.md` (from planner)
  - `planning/deck-visual-design.md` (from visual designer)
- `diagrams/`: Diagram sources (`.mmd`/`.mermaid` for Mermaid, `.excalidraw` for Excalidraw) created by the visual design phase.
- `.marp/assets/`: Generated images (e.g., Mermaid â†’ PNG) and local images used by the deck.
- `.marp/scripts/`: Helper scripts used by Taskfile tasks (e.g., Mermaid asset validation).
- `.marp/references/deck-template.md`: Authoritative layout and template rules used by the generator.
- `output/`: Exported artifacts (PDF/PPTX) written by build tasks.
- `samples/`: Optional examples (empty by default).
- `AGENT.md`: Implementation rules for building slides with this template.
- `Taskfile.yml`: Build tasks (preview, pdf, pptx, mermaid, etc.).

## How to generate a deck

1. **Prepare your content** â€” Write a description of what you want to present and save it as `definition/deck-definition.md`. This is the file the installed skills expect as the starting point. A sample file (`definition/deck-definition.md.sample`) is provided for reference.
2. **Run the workflow** â€” Use the `/marp-deck-workflow` skill to follow through all phases (planning, visual design, generation, consistency check) and produce the final deck.

## Build & preview
All tasks run Marp CLI via Docker. Start Docker Desktop first.

```
task preview                             # live preview at http://localhost:8080
task pdf                                 # export versioned PDF to output/
task pptx                                # export versioned PPTX to output/
task mermaid                             # build diagrams/*.mmd into assets/*.png
task mermaid-file FILE=diagrams/foo.mmd  # build a single diagram
task mermaid-check                       # validate Mermaid outputs + aspect ratios
task mermaid-check-file FILE=diagrams/foo.mmd  # validate one diagram output
task excalidraw                                    # export diagrams/*.excalidraw into images/*.png
task excalidraw-file FILE=diagrams/foo.excalidraw  # export a single Excalidraw file
task artifacts                           # list generated artifacts
task clean                               # remove output/
```

Defaults/pins:
- Marp CLI image: `marpteam/marp-cli:v4.2.2`
- Mermaid CLI image: `minlag/mermaid-cli:11.12.0`
- Mermaid theme config: `.marp/references/mermaid-config.json` (aligned to `.marp/themes/deck-theme.css`)
- Excalidraw export: `.marp/scripts/render-excalidraw.js` via node + `@napi-rs/canvas` (no Docker; `npm install` auto-runs on first use)
- Preview port: `8080`

## Deck metadata (front matter)
Required at the top of `slides/deck.md`:

```

marp: true
title: <presentation title>
author: <author>
theme: deck-theme
paginate: false
html: true
size: 16:9
---
```

## Diagrams and images
- Place diagram sources in `diagrams/` as `.mmd`/`.mermaid` (Mermaid) or `.excalidraw` (Excalidraw).
- Generate Mermaid images with `task mermaid` (writes to `images/`).
- Generate Excalidraw images with `task excalidraw` (writes to `images/`).
- From slides, reference assets with paths relative to `slides/`: `![](../.marp/assets/<name>.png)` or `![](../images/<name>.png)`.
- One image per slide unless a grid is explicitly planned by the visual design.

## Template rules and allowed layouts
See `.marp/references/deck-template.md` for the canonical mapping of layouts/classes and constraints (section slides, twoâ€‘column patterns, grids, lifecycle, ranked lists, etc.). Prefer existing classes; extend CSS only if strictly necessary.

## Endâ€‘toâ€‘end deck workflow (skills)
This repository includes a set of skills that drive a deterministic, multiâ€‘phase pipeline. Use the orchestrator as the single entry point, or run phases manually.

1) Planner â€” `marp-deck-planner`
	- Input: `definition/deck-definition.md`
	- Output: `planning/deck-plan.md` (slide sequence, structure, content intent; no diagrams)

2) Visual designer â€” `marp-deck-visual-designer`
	- Input: `planning/deck-plan.md`
	- Outputs: `planning/deck-visual-design.md`, `diagrams/*.mmd`, `diagrams/*.excalidraw`, `assets/*.png`, `images/*.png`
	- Rules: preserve slide order; materialize only visuals that were planned

3) Deck generator â€” `marp-deck-gen`
	- Inputs: `definition/deck-definition.md`, `planning/deck-plan.md`, `planning/deck-visual-design.md`, `.marp/references/deck-template.md`
	- Outputs: `slides/deck.md`, `slides/deck-notes.md`
	- Behavior: deterministic render; no new content or slides

4) Consistency checker â€” `marp-deck-checker`
	- Inputs: definition, plan, visual design, deck, template rules
	- Output: `qa/deck-consistency-report.md` (the `qa/` folder will be created if needed)

Oneâ€‘shot orchestrator â€” `marp-deck-workflow`
- Purpose: run 1â†’4 in order; stop on mismatches; produce all artifacts deterministically.
- Preconditions: `definition/deck-definition.md` and `.marp/references/deck-template.md` must exist.
- Expected outputs: `planning/deck-plan.md`, `planning/deck-visual-design.md`, `diagrams/*.mmd`, `assets/*.png`, `slides/deck.md`, `slides/deck-notes.md`, `qa/deck-consistency-report.md`.

## Notes
- The theme name in `slides/deck.md` must match `@theme deck-theme` in `.marp/themes/deck-theme.css`.
- Keep asset paths relative to `slides/` (e.g., `../.marp/assets/...`).
- Frontpage (`<!-- _class: title -->` or `<!-- _class: title cover [variant] -->`) and closing (`<!-- _class: closing -->`) must remain intact; user slides are inserted between them.
